# Аудит `moex_store`: сомнительные места и план оптимизации

Дата анализа: 2026-05-08.

Я смотрел проект как библиотеку для Backtrader, которая синхронным API (`getdata`) ходит в MOEX ISS через `aiohttp` и `aiomoex`, кэширует свечи в CSV и отдельно умеет работать со справочниками фьючерсов.

Главный вывод: проблема "MOEX часто недоступна, а через VPN стабильно" очень похожа не на одну ошибку, а на сочетание нескольких факторов:

1. DNS-подмена есть, но применяется не ко всем запросам.
2. Большая часть реальных запросов не имеет устойчивых ретраев и явных таймаутов.
3. Даже при `read_from_file=True` код все равно лезет в MOEX до чтения кэша.
4. `aiohttp` патчится глобально и повторно при каждом создании `MoexStore`, что делает поведение сетевого слоя трудно предсказуемым.
5. В фьючерсных методах много последовательных сетевых запросов, каждый из которых может упасть.

Ниже предложения идут в порядке, в котором я бы их разбирал.

## 1. Сначала починить сетевой слой, а не точечно менять DNS

Файлы: `moex_store/store.py`, `moex_store/dns_client.py`, `moex_store/patch_aiohttp.py`, `moex_store/futures.py`.

Сейчас сетевой код размазан:

- `store.py` напрямую создает `aiohttp.ClientSession`;
- `futures.py` тоже напрямую создает `aiohttp.ClientSession`;
- `aiomoex` переопределяется через `patch_aiohttp.py`;
- `dns_client.py` пытается подменять DNS только внутри `aiomoex`;
- SSL-контекст внедряется через глобальный monkey-patch `aiohttp`.

Что плохо:

- Невозможно гарантировать, что все запросы используют одни и те же DNS, SSL, таймауты и ретраи.
- Если падает прямой запрос из `store.py` или `futures.py`, DNS-клиент из `dns_client.py` может вообще не участвовать.
- При отладке непонятно, какой именно механизм сейчас отвечает за соединение.

Предложение:

Создать один внутренний модуль, например `moex_store/http_client.py`, который будет отвечать за все HTTP-запросы к MOEX:

- единый `ssl_context` на `certifi`;
- явные `ClientTimeout`;
- единый `TCPConnector`;
- настраиваемые DNS-серверы;
- fallback на системный DNS;
- fallback на альтернативные публичные DNS;
- опциональное принудительное IPv4;
- retry с exponential backoff и jitter;
- единое логирование ошибок.

Пример параметров, которые стоит поддержать в `MoexStore.__init__`:

```python
MoexStore(
    write_to_file=True,
    read_from_file=True,
    dns_servers=("77.88.8.8", "1.1.1.1", "8.8.8.8"),
    force_ipv4=True,
    request_timeout=30,
    connect_timeout=10,
    max_retries=5,
    retry_delay=1.0,
    trust_env=True,
)
```

Почему `force_ipv4=True` важен: если у провайдера или ОС ломается IPv6-маршрут до `iss.moex.com`, то DNS вроде бы резолвится, но соединение нестабильно. VPN часто "лечит" это тем, что меняет маршрутизацию и DNS. В `aiohttp.TCPConnector` можно задать `family=socket.AF_INET`, чтобы исключить IPv6 как источник проблемы.

## 2. Убрать глобальный monkey-patch `aiohttp`

Файл: `moex_store/store.py`, строки 32-55.

Сейчас `MoexStore.__init__` вызывает `self.apply_ssl_patch()`, а этот метод заменяет:

- `aiohttp.ClientSession.__init__`;
- `aiohttp.TCPConnector.__init__`.

Это опасно по двум причинам.

Первая: патч глобальный. Он меняет поведение `aiohttp` во всем Python-процессе, включая чужой код пользователя.

Вторая: патч применяется при каждом создании `MoexStore`. В `test.py` внутри цикла создается новый `MoexStore()` для каждого контракта. Значит `ClientSession.__init__` и `TCPConnector.__init__` оборачиваются повторно. Каждая новая обертка ссылается на предыдущую уже пропатченную версию. Это может приводить к цепочке вложенных вызовов и трудноуловимым побочным эффектам.

Предложение:

- Не менять `aiohttp` глобально.
- Создавать `ssl_context = ssl.create_default_context(cafile=certifi.where())`.
- Передавать его явно в `aiohttp.TCPConnector(ssl=ssl_context, ...)`.
- Все `ClientSession` создавать через одну фабрику/контекстный менеджер проекта.

## 3. Сделать DNS-настройку настоящей и единой

Файлы: `moex_store/dns_client.py`, `moex_store/patch_aiohttp.py`.

Сейчас DNS-подмена работает так:

- `patch_aiohttp.py` заменяет `aiomoex.client.ISSClient` на `DNS_ISSClient`;
- `DNS_ISSClient.get()` пробует DNS `77.88.8.1` и `8.8.8.8`;
- внутри метода меняется приватное поле `self._session._connector`.

Проблемы:

- Прямые запросы через `aiohttp` в `store.py` и `futures.py` этот DNS-клиент не используют.
- Использование приватного поля `_connector` ненадежно: это внутренняя деталь `aiohttp`.
- Коннектор закрывается в `finally`, но сессия продолжает жить. Для многостраничных запросов `aiomoex.get_all()` это может давать странное поведение.
- DNS-серверы жестко зашиты в код, их нельзя поменять из `MoexStore`.
- Нет системного DNS как fallback.
- Нет Cloudflare/Quad9 как вариантов.

Предложение:

- Удалить замену `aiomoex.client.ISSClient` как основной механизм.
- Использовать обычный `aiomoex`, но передавать ему `ClientSession`, созданную с нужным `TCPConnector`.
- DNS-серверы сделать параметром `MoexStore` и/или переменной окружения.
- Порядок fallback сделать таким:
  1. системный DNS;
  2. DNS из настроек пользователя;
  3. безопасный набор по умолчанию.

Отдельно проверить набор DNS для РФ. Сейчас `8.8.8.8` может быть недоступен или нестабилен у некоторых провайдеров. Я бы начал с `77.88.8.8`, `1.1.1.1`, `8.8.8.8`, но оставил это настройкой, а не константой в коде.

## 4. Исправить явную ошибку в `dns_client.py`

Файл: `moex_store/dns_client.py`, строка 40.

Сейчас при HTTP-ошибке выполняется:

```python
raise ISSMoexError("Неверный url", result.url) from err
```

Но `result` в этот момент еще не определен. Должно быть что-то вроде:

```python
raise ISSMoexError("Неверный url", response.url) from err
```

Эта ошибка маскирует настоящую причину отказа MOEX. Вместо нормального сообщения о HTTP-статусе можно получить вторичную ошибку `UnboundLocalError`.

## 5. Сделать кэш реально офлайн-режимом

Файл: `moex_store/store.py`, строки 116-135.

Сейчас порядок такой:

1. Нормализуются даты.
2. Вызывается `_validate_inputs(...)`.
3. Только потом проверяется CSV-файл в `files_from_moex`.

Проблема: `_validate_inputs(...)` делает сетевые запросы:

- `get_instrument_info`;
- `get_history_intervals`.

Значит даже если CSV уже есть на диске и `read_from_file=True`, библиотека все равно может упасть из-за недоступности MOEX до чтения локальных данных.

Это, на мой взгляд, один из главных практических багов проекта.

Предложение:

- Сначала валидировать только локальные вещи: типы, даты, `tf`, путь к файлу.
- Потом проверять наличие кэша.
- Если кэш найден и `read_from_file=True`, читать его без сетевых запросов.
- Сетевую валидацию делать только если данных на диске нет или пользователь явно просит `validate_remote=True`.

Пример целевого поведения:

```python
store = MoexStore(read_from_file=True, offline=True)
data = store.getdata("AFLT", "2023-01-01", "2024-01-01", "1h", "AFLT")
```

В таком режиме код вообще не должен ходить в MOEX, если CSV найден.

## 6. Убрать сетевые запросы из проверки "активный фьючерс" при чтении CSV

Файл: `moex_store/store.py`, строки 124-130.

Даже когда CSV найден, для фьючерса вызывается:

- `self.futures.get_asset_code(sec_id)`;
- `self.futures.get_active_contract(...)`.

Обе функции ходят в MOEX. То есть кэш снова не спасает от недоступной биржи.

Предложение:

- Проверку "активный фьючерс, кэш мог устареть" сделать опциональной.
- По умолчанию не делать сетевых запросов при чтении CSV.
- Если нужна проверка свежести, делать ее отдельным методом: `store.check_cache_staleness(...)`.

## 7. Добавить ретраи не только в `_check_connection`, а во все запросы

Файл: `moex_store/store.py`, строки 73-114, 233-277, 279-335; `moex_store/futures.py`, строки 240-301.

Сейчас ретраи есть только в `_check_connection`. Но реальная загрузка может упасть после успешной проверки:

- `_check_connection()` успешно сходил в `/engines.json`;
- потом `get_instrument_info()` упал;
- или `get_history_intervals()` упал;
- или `get_market_candles()` упал на одной из страниц.

Проверка доступности перед запросом не заменяет ретраи вокруг самого запроса.

Предложение:

- Сделать одну функцию `request_json_with_retries(...)`.
- Для `aiomoex` обернуть вызовы в retry на уровне операции.
- Ретраить только временные ошибки:
  - `ClientConnectorError`;
  - `ServerDisconnectedError`;
  - `ClientOSError`;
  - `asyncio.TimeoutError`;
  - HTTP `429`, `500`, `502`, `503`, `504`.
- Не ретраить постоянные ошибки:
  - `400`;
  - `401`;
  - `403`;
  - `404`, если URL корректный, но инструмент реально не найден.

## 8. Добавить явные таймауты

Файлы: `store.py`, `futures.py`, `dns_client.py`.

Сейчас `aiohttp.ClientSession()` почти везде создается без явного `ClientTimeout`. При проблемах с DNS, маршрутом или зависшим сервером пользователь может долго ждать или получить нестабильное поведение.

Предложение:

```python
timeout = aiohttp.ClientTimeout(
    total=60,
    connect=10,
    sock_connect=10,
    sock_read=30,
)
```

Для больших исторических загрузок `total` можно делать больше, но `connect` и `sock_connect` должны оставаться короткими. Если проблема именно в доступности MOEX, важно быстро понять, что конкретный endpoint сейчас не отвечает, и перейти к следующей попытке/DNS.

## 9. Не использовать `time.sleep()` внутри async-кода

Файл: `moex_store/store.py`, строки 107, 112, 297.

Внутри async-функций используется `timer.sleep(...)`. Это блокирует event loop.

Предложение:

- В `_check_connection()` заменить на `await asyncio.sleep(...)`.
- В `_get_candles_history()` убрать `timer.sleep(0.05)` или заменить на `await asyncio.sleep(0.05)`.

Это не главная причина сетевых падений, но это ухудшает поведение при будущей параллельной загрузке нескольких инструментов.

## 10. Перестать вызывать `asyncio.run()` на каждый мелкий запрос

Файлы: `store.py`, `futures.py`.

Сейчас публичные методы синхронные, но внутри много `asyncio.run(...)`:

- `store.py`: строки 132, 133, 186, 202;
- `futures.py`: строки 30, 147, 190, 198.

Проблемы:

- Каждый вызов создает и закрывает event loop.
- Это ломается внутри уже работающего event loop, например в Jupyter, FastAPI, GUI-приложениях.
- Нельзя нормально переиспользовать `ClientSession`.
- Нельзя эффективно загружать несколько запросов параллельно.

Предложение:

- Сделать async-ядро: `async_get_data`, `async_get_instrument_info`, `async_get_history_intervals`.
- Синхронный `get_data()` оставить как тонкую обертку для Backtrader.
- Внутри одной операции использовать один event loop и одну HTTP-сессию.

## 11. Переиспользовать одну `ClientSession` на операцию

Файлы: `store.py`, `futures.py`.

Сейчас почти каждый метод создает свою сессию:

- `get_instrument_info`;
- `get_history_intervals`;
- `_get_candles_history`;
- `_get_all_futures`;
- `_get_futures_info`;
- `_get_futures_stat`.

Это дорого и увеличивает шанс сетевых сбоев.

Предложение:

Для одного вызова `get_data()` создать одну сессию:

1. проверить инструмент;
2. получить интервалы;
3. скачать свечи;
4. закрыть сессию.

Для фьючерсных справочников тоже использовать одну сессию на всю операцию.

## 12. Не закрывать чужую сессию в `futures.py`

Файл: `moex_store/futures.py`, строки 244 и 286.

Сейчас используется паттерн:

```python
async with session or aiohttp.ClientSession() as session:
```

Если в функцию когда-нибудь передадут уже созданную сессию, `async with session` закроет ее при выходе. Это неожиданно для вызывающего кода.

Предложение:

Разделить случаи:

```python
if session is None:
    async with make_session() as session:
        return await ...
return await ...
```

## 13. Исправить параметр `name`

Файл: `moex_store/store.py`, строки 116 и 174-176.

В сигнатуре:

```python
def get_data(..., name=None):
```

Но в `_validate_inputs`:

```python
if not isinstance(name, str):
    raise ValueError(...)
```

То есть значение по умолчанию `None` сразу невалидно. README также описывает `name` как необязательный параметр.

Предложение:

Разрешить `None`:

```python
if name is not None and not isinstance(name, str):
    raise ValueError(...)
```

## 14. Явно обрабатывать пустой ответ по свечам

Файл: `moex_store/store.py`, строки 327-338.

Если `_get_candles_history()` вернет `None`, `_make_df(None, ...)` создаст пустой DataFrame и потом упадет с ошибкой про отсутствующие колонки. Это маскирует настоящую причину: свечи не найдены или запрос не выполнен.

Предложение:

- Если `moex_data is None` или пустой список, бросать понятное исключение до `_make_df`.
- В сообщении указать `sec_id`, `tf`, `fromdate`, `todate`, `market`, `engine`, `board`.

## 15. Разбить длинные загрузки свечей на чанки

Файл: `moex_store/store.py`, строки 279-335.

Сейчас большой диапазон дат отдается в один вызов `aiomoex.get_market_candles(...)`, а `aiomoex` внутри сам ходит по страницам. Если один запрос из цепочки упадет, вся загрузка может сорваться.

Предложение:

- Для минутных данных грузить диапазон частями, например по месяцу или кварталу.
- После каждого успешного чанка сохранять промежуточный результат.
- При повторном запуске продолжать с последней успешно загруженной даты.

Это особенно важно для плохой сети: лучше потерять один чанк и повторить его, чем начинать весь год минуток заново.

## 16. Улучшить прогресс загрузки

Файл: `moex_store/store.py`, строки 292-307 и 372-393.

Сейчас прогресс-бар основан на эмпирической оценке времени, а не на реальном количестве страниц или дат. Если сеть зависла, прогресс может выглядеть странно.

Предложение:

- При чанковой загрузке показывать прогресс по датам/чанкам.
- Для `aiomoex.get_all()` можно ориентироваться на cursor, если оставить прямую работу с ISSClient.

## 17. Оптимизировать справочники фьючерсов

Файл: `moex_store/futures.py`.

Особенно узкое место: `get_all_active_futures`, строки 190-228.

Сейчас:

1. Загружается список активных фьючерсов.
2. Для каждого фьючерса отдельным запросом вызывается `_get_futures_info`.
3. Каждый запрос идет последовательно.
4. Каждый запрос создает свою сессию.

Если активных контрактов много, это десятки или сотни сетевых запросов. При нестабильном доступе к MOEX это будет часто падать.

Предложение:

- Переиспользовать одну сессию.
- Выполнять запросы параллельно с лимитом, например 5-10 одновременных запросов.
- Кэшировать справочную информацию фьючерсов хотя бы на день.
- Проверить, нельзя ли нужные поля получить из одного endpoint без дополнительных запросов.

## 18. Исправить `get_history_stat(show_table=True)`

Файл: `moex_store/futures.py`, строки 166-174.

Сейчас если `show_table=True`, таблица создается, но после этого метод не возвращается из успешной ветки и доходит до сообщения:

```python
Биржа не вернула статистику...
```

То есть пользователь может получить таблицу и одновременно ложное сообщение об ошибке.

Предложение:

После `_display_table(...)` добавить явный `return None` или вернуть `history_stat`, если хочется сохранить данные.

## 19. Не открывать браузер из библиотечного метода без явного запроса

Файл: `moex_store/futures.py`, строки 321-334.

`_display_table` всегда вызывает:

```python
webbrowser.open(...)
```

Проблемы:

- ломает headless-окружения;
- плохо для тестов;
- неожиданно для библиотечного кода;
- требует GUI.

Предложение:

- Добавить параметр `open_browser=True`.
- Для библиотечного API лучше по умолчанию сохранять файл и возвращать путь.
- Открытие браузера оставить как явную опцию.

## 20. Исправить default-аргументы с `datetime.today()`

Файл: `moex_store/futures.py`, строки 63, 93, 303.

Сейчас `datetime.today()` вычисляется один раз при загрузке модуля, а не при каждом вызове функции.

Пример:

```python
def get_active_contract(self, asset, date=datetime.today()):
```

Если процесс живет долго, дата по умолчанию устареет.

Предложение:

```python
def get_active_contract(self, asset, date=None):
    if date is None:
        date = datetime.today()
```

То же самое для `get_contracts_between` и `_get_active_contract`.

## 21. Переделать пересчет 5/15/30 минут на pandas resample

Файл: `moex_store/tf.py`.

Сейчас пересчет таймфрейма делается вручную:

- данные мутируются на месте (`item['minutes_value'] = ...`);
- группировка через `itertools.groupby` корректна только если входные данные уже отсортированы;
- `fill_missing_time` начинает с начала часа первого бара, а не с биржевой сессии или правильной границы таймфрейма;
- пустые бары создаются с ценами `-1`, потом фильтруются.

Это хрупко. Ошибка в агрегации свечей хуже сетевой ошибки: код будет работать, но данные будут неверными.

Предложение:

- Переводить список свечей в DataFrame.
- Индексировать по `begin`.
- Использовать `resample("5min")`, `resample("15min")`, `resample("30min")`.
- Агрегация:
  - `open`: first;
  - `high`: max;
  - `low`: min;
  - `close`: last;
  - `volume`: sum;
  - `value`: sum.
- Пустые интервалы удалять по отсутствующему `open`.

Так будет проще, быстрее и надежнее.

## 23. Согласовать зависимости `pyproject.toml` и `requirements.txt`

Наблюдения:

  - `certifi`;

## 25. Добавить нормальное логирование вместо `print`

Файлы: почти весь пакет.

Сейчас много `print(...)`. Для библиотеки лучше использовать `logging`.

Предложение:

- Создать logger `logging.getLogger("moex_store")`.
- Информационные сообщения писать в `logger.info`.
- Сетевые повторы в `logger.warning`.
- Детали исключений в `logger.debug(..., exc_info=True)`.
- Для пользователя оставить параметр `verbose=True/False`.

Это сильно упростит диагностику случаев "через VPN работает, без VPN нет": можно будет увидеть endpoint, DNS, номер попытки, тип ошибки и длительность.

## 26. Уточнить и унифицировать исключения

Сейчас часть ошибок:

- печатается и возвращает `None`;
- часть превращается в `ValueError`;
- часть в `MOEXConnectionError`;
- часть остается `aiohttp.ClientError`;
- часть может стать случайным `KeyError` или `TypeError`.

Предложение:

Ввести небольшую иерархию:

```python
class MoexStoreError(Exception): ...
class MoexNetworkError(MoexStoreError): ...
class MoexUnavailableError(MoexNetworkError): ...
class MoexInstrumentNotFound(MoexStoreError): ...
class MoexNoDataError(MoexStoreError): ...
class MoexCacheError(MoexStoreError): ...
```

Тогда пользователь сможет отдельно ловить сетевые ошибки и, например, переходить на кэш.

## 29. Предлагаемый порядок работ

## 30. Что, вероятнее всего, даст быстрый эффект

Если нужна самая короткая практическая дорожная карта, я бы начал с четырех изменений:

1. Перенести проверку CSV-кэша выше `_validate_inputs`.
2. Отключить сетевую проверку активного фьючерса при чтении CSV.
3. Заменить глобальный SSL/DNS monkey-patch на явный `TCPConnector` с `AsyncResolver`, `ClientTimeout`, `family=AF_INET` и retry.
4. Обернуть все реальные запросы к MOEX в общий retry, а не только `_check_connection`.

Это должно напрямую ударить по текущей боли: "без VPN часто отваливается, с VPN работает".
